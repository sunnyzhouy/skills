对「$ARGUMENTS」执行 Deep Research Pipeline 调研。

## 方法论

基于 OpenAI/Perplexity Deep Research 架构 + Meta CoVe 验证机制。不是固定 3 轮自我辩论（LLM 自我纠错不可靠），而是动态 7 步管道：范围界定 → 广域搜索 → 中间综合 → 缺口补充 → 反面搜索 → 关键验证 → 最终报告。

**整个过程你自己跑，中间不要问我，最后一次性呈现结果。**

---

## Step 1：Scope — 范围界定

把课题拆成 3-5 个可独立搜索的子问题。每个子问题必须足够具体，能直接变成搜索 query。

```
课题：$ARGUMENTS
子问题：
  Q1: [具体问题]
  Q2: [具体问题]
  Q3: [具体问题]
  Q4: [如有必要]
```

如果课题涉及市场/产品方向，调用 `/market-research $ARGUMENTS`。
如果课题涉及竞品对标，调用 `/competitive-brief $ARGUMENTS`。

## Step 2：Broad Search — 广域搜索

对每个子问题执行 2-3 次搜索，覆盖三个视角：

| 视角 | 搜索策略 | 示例 query |
|------|---------|------------|
| 学术/论文 | 搜 arxiv, scholar, 顶会 | `"$ARGUMENTS" research paper 2024 2025 2026` |
| 工业/产品 | 搜 GitHub, HN, 产品案例 | `"$ARGUMENTS" implementation open source` |
| 第一性原理 | 搜基础原理、替代路径 | `"$ARGUMENTS" alternative approach first principles` |

**并行搜索**：多个子问题的搜索可以并行发起。目标：每个子问题至少找到 3 个高质量来源。

## Step 3：Interim Synthesis — 中间综合

基于 Step 2 的搜索结果，生成：

1. **初步发现**：每个子问题的 2-3 句核心结论
2. **知识缺口清单**：显式列出"我还不知道什么"
   ```
   已知：[列出已有确定答案的问题]
   存疑：[列出有矛盾信息的问题]
   未知：[列出完全没找到信息的问题]
   ```
3. **初步方案雏形**：基于已知信息的 3-4 个方向（此时不收敛，允许多于 3 个）

## Step 4：Gap-Fill Search — 缺口补充

针对 Step 3 的"存疑"和"未知"做定向搜索：

- 每个缺口 1-2 次搜索
- 搜索 query 要比 Step 2 更具体、更窄
- 如果搜不到，标注"信息不可得"并降低该维度的置信度

如果课题涉及当前项目，在此步骤中结合代码库分析可行性：
- 现有架构能否支撑？
- 有没有现有代码已经部分解决了？

## Step 5：Contrarian Search — 反面搜索

这是替代"自我挑战"的关键步骤——用外部证据挑战，不用自我反思。

对 Step 3 的每个方案方向，搜索：

1. **失败案例**：`"[方案关键词] failure post-mortem"` `"[方案关键词] didn't work"`
2. **反对意见**：`"[方案关键词] criticism"` `"why not [方案关键词]"`
3. **替代方案**：`"[方案关键词] vs [替代方案]"` `"[方案关键词] alternative"`

对找到的反面证据，判断：
- 反驳成立 → 降级或淘汰该方案
- 反驳有条件 → 标注适用边界
- 反驳不成立 → 方案信心增强，记录为什么反驳无效

## Step 6：CoVe Verify — 关键声明验证

从所有方案中提取 3-5 个最关键的事实声明（不验证观点，只验证事实），执行 Chain-of-Verification：

```
声明 → 生成验证问题 → 搜索回答验证问题 → 标注 [已验证 ✓ / 存疑 ? / 反驳 ✗]
```

例如：
- 声明："OpenAI Deep Research 使用 RL 端到端训练"
- 验证问题："OpenAI Deep Research System Card 是否提到 RL 训练？"
- 搜索验证 → ✓ 已验证（来源：OpenAI System Card PDF）

未通过验证的声明必须在最终报告中标注。

## Step 7：Final Report — 最终报告

收敛为 3 套方案。不要给我 6 套。如果有第 4 套有价值的，合并到最接近的方案里。

## 搜索工具优先级

**必须获取互联网最新内容**，不允许仅依赖训练数据。按以下优先级选择搜索工具：

1. **WebSearch**（首选，快速获取搜索结果列表）
2. **Exa MCP**（`/exa-search`）——神经语义搜索，适合模糊/概念性查询
3. **WebFetch**——直接抓取已知 URL 的完整内容
4. **`/browse`**——无头浏览器，适合需要渲染 JS 的页面、交互式内容、搜索引擎直接操作（Google/Bing）、PDF 下载
5. **Playwright MCP**——同上，如果 `/browse` 不可用时的备选
6. **DeepWiki MCP**——开源项目文档和架构
7. **Context7 MCP**——框架/库的最新文档

**工具选择原则**：静态页面用 WebFetch，动态页面/需要点击搜索的用 `/browse`，语义搜索用 Exa，技术文档用 Context7/DeepWiki。多个工具并行使用以加速。

## 并行模式（可选）

使用 `/3r --parallel [课题]` 时，Step 2 改为 3 个独立 subagent 并行执行：

```
Agent 1: 学术/论文路径
Agent 2: 工业/产品路径
Agent 3: 第一性原理路径
```

三个 agent 互相看不到对方的输出。Step 3 由主 agent 综合三份独立报告。
其余步骤不变。

适用场景：战略决策、争议大的技术选型、需要最大化观点多样性的课题。
代价：3x subagent 调用成本。

## 约束

- 全过程自己跑完，不中途问我。最后一次性呈现结果
- 每步必须实际搜索互联网，不能跳过
- 引用来源必须标注 URL。没有来源的论断标注"推断"
- 收敛到 3 套方案，不多不少
- 如果课题涉及当前项目，Step 4 中结合代码库现状分析
- 如果所有搜索工具都不可用（极端情况），标注"离线模式：仅训练数据"并将所有置信度降一档
